## 1. Bugs / correctness hazards

### 1.1 Duplicate migration version `20260523000000` — **known cosmetic collision, safe to leave**

Two migrations share the same version timestamp:

```
supabase/migrations/20260523000000_client_companies.sql
supabase/migrations/20260523000000_resume_score_rename.sql
```

Supabase tracks applied migrations by the leading version string, so a shared version is only a
hazard _at apply time_. Both migrations have already been applied successfully to all
environments (confirmed 2026-05-31), so that moment has passed. The two are also
content-independent — a column rename + a new table — so they apply cleanly in either order, and
even a fresh `supabase db reset` from these files reproduces the same schema. The usual
duplicate-version danger (ambiguous ordering) does not materialize here.

**Decision: leave it.** Renaming now would be the riskier move — the CLI would see the new
version string as an unapplied migration and try to re-run `alter table resumes rename column
ats_score to resume_score`, which fails (already renamed), and you'd then have to hand-reconcile
`supabase_migrations.schema_migrations`. Only revisit if `supabase migration list` ever surfaces
a local/remote mismatch on this timestamp.

### 1.2 No other correctness bugs found in spot-checks

- `src/lib/plan.ts` `canSeeJobTier()` matches the SQL `can_see_job_tier()` exactly
  (tier_2 ⇒ plan_1/2/3, tier_3 ⇒ plan_3). No client/DB gating drift.
- `ats_score → resume_score` rename is clean — zero lingering `ats_score`/`atsScore`
  references in `src/`.
- Onboarding soft-gate is correctly wired: `/job-board` + `/job-board/[id]` `redirect()` to
  `/onboarding/preferences` when `onboarding_completed_at` is null; `/dashboard` shows a
