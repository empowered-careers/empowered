How the Blueprint thinks:
logic, scoring & who reads it

The Career Identity Blueprint™ isn't just a results screen — it's a reusable candidate-intelligence layer. This explains how 30 answers become normalized axes, and how three different systems consume them.

Companion to docs/done/career-blueprint-integration.md · prototype UI in docs/prototypes/career-identity-blueprint.html

01 · The big picture

One assessment, three consumers

A candidate answers 30 multiple-choice questions. We don't store "answers" as the product — we store a set of normalized 0–100 dimensions. Those dimensions are the source of truth that everything downstream reads.

The intrinsic voice + motivation signal tunes the resume rewrite and LinkedIn audit so the output sounds like the actual person.

3. Company-Culture Match

A future company profile is scored on the same axes. Matching becomes a similarity calc — preference vs reality, trait vs culture.

Reads: culture_axes (the shared vocabulary).

Why this matters: if we stored ad-hoc, screen-specific scores (like the throwaway prototype did), each new consumer would need its own re-derivation. By committing to one canonical axis vocabulary, the candidate side and the company side speak the same language, and the LLMs get clean structured context for free.

Key inversion vs the prototype: the demo computed display copy directly and the "scores" were a byproduct. 04 · Scoring mechanics

How weights become 0–100

There are three axis shapes, each scored a little differently:

Spectrum axes (e.g. structure_vs_autonomy)

Each option contributes non-negative votes to one pole or the other (two separate counters per axis — e.g. autonomy and structure). The axis is centered at 50 and shifted by a fixed coefficient times the difference, then clamped to 0–100. 50 = neutral / no signal.

# two non-negative counters per axis, summed across options autonomy = Σ optionWeight(autonomy) structure = Σ optionWeight(structure) score = clamp( 50 + 10 × (autonomy − structure), 0, 100 ) # same shape for people↔analytical, vision↔execution, external↔internal # pace_sustainability adds a burnout_risk term: pace = clamp( 50 + 10·sustainability − 10·intensity + 8·burnout_risk, 0, 100 )

The coefficient (10) was picked so a strongly expressed preference (4-vote net) lands at ~90 and a balanced answer set stays near 50. Pre-flight sanity-checked against synthetic profiles.

Distribution axes (e.g. company_stage_fit)

Each option adds points to one category. We keep the whole normalized distribution (percentages summing to 100), not just the winner — because a company match needs the full shape. "60% startup / 30% growth" matches very differently from "100% startup".

# tally points per category, then normalize to a share of total startup:24 growth:12 enterprise:4 mission:0 # raw tallies → { startup: 60, growth: 30, enterprise: 10, mission: 0 }

Categorical axes (e.g. leadership_style)

Tally points per category; the top category is the result (with the runner-up retained for tie-breaks / nuance). This is how the 5 leadership styles and 4 communication styles are picked.

Determinism guarantee: the same 30 answers always produce the same axes and the same archetype. There is no randomness and no model call. (The prototype's cosmetic "Analyzing…" spinner is just a transition — the math is sub-millisecond.)

06 · Where it lives

Two homes: queryable axes + display blob

Structured axes go where matching can query them efficiently; the rich display blob lives with the response record.

candidate_scores
one row per profile · the matching home
  • culture_axes jsonb — all §03 axes (0–100 + distributions). GIN-indexed. Read by the matcher & LLMs.
  • role_clarity / values / strengths / leadership / impact — the 5 P1 dims, derived from the axes
  • overall_score
  • mindset / communication — left null (Phase 2)
assessment_responses
one row per (profile, assessment) · upsert on retake
  • responses jsonb — the raw 30 answers
  • result jsonb — the display blob, incl. embedded voiceProfile
  • archetype text — denormalized for fast lists/sharing
  • score · completed_at
07 · Consumer impact

What each system does with the data

✍️ Resume + LinkedIn personalization fast-follow

The existing scoreResume() / scoreLinkedIn() LLMs are extended with an optional voiceProfile argument, injected into the per-request user message (the cached rubric system prompts stay untouched). When absent, behavior is unchanged — purely additive.

Effect: an Executive Strategist with vision/builder wiring gets resume bullets led by strategic impact and a concise, authoritative LinkedIn About; a Trusted Advisor with human-led wiring gets warmer, relationship-centered phrasing. Same facts, voice tuned to who they are.

🧭 Company-culture match future

The future company profile is scored on the identical axis keys. The matcher reads both sides from structured storage — no re-derivation:

# preference axes vs company reality fit_pref = similarity(candidate.company_stage_fit, company.stage_profile) + similarity(candidate.culture_values, company.culture_profile) # trait axes vs company culture archetype fit_trait = compatibility(candidate.traits, company.culture_archetype) match = w1·fit_pref + w2·fit_trait # matcher tunes the weights

Keeping preference and trait axes separate is what lets the matcher weight "do they want this environment" independently from "will their wiring fit the culture."
